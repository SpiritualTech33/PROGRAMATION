What are primary and foreign keys in SQLite context?

Before talking about primary's and foreign keys, the first important question we must should be doing is, what is a key inside the database context?
Down this context, a key is a unique identificatory of a particular object in the data base. We can think of this in the reality itself.
	Your fingerprint is a key (Most of the documentation process are made with your fingerprint identification.)
	Your DNA is a key, you are the only human in the world with that particular configuration.
	Your phone number is a key, it connects with your contact.
	Your memories and psique are a key, you are the only one person in the world that is actually YOU, with your experiences, your memories, your dreams and your goals.

Keys are how we establish identity in structured reality.

With this said, now we can talk about keys in SQLite.

# **Primary Key**

As we said before, a key it's somethin than identifies a particular object and makes a relationship between that object and the key. 
It's a column (or a combination of columns) that identifies one row in a table.
Two rows can't have the same primary key, and the primary can't be NULL or EMPTY, must be filled with something (a number like an ID, a name, or an idea, but something must be identify that row.) A primary key is the "self" of a database row.


**Example in Reality**
Let's return to our Philosophers table:

```sql
CREATE TABLE philosophers ( 
philosopher_id INTEGER PRIMARY KEY, 
name TEXT NOT NULL, 
birth_year INTEGER, 
death_year INTEGER, 
school_of_thought TEXT );
```


| philosopher_id | name      | birth_year | death_year | school_of_thought |
|----------------|-----------|------------|------------|-------------------|
| 1              | Plato     | -428       | -348       | Idealism          |
| 2              | Kant      | 1724       | 1804       | Transcendental    |
| 3              | Socrates  | -470       | -399       | Classical         |
| 4              | Descartes | 1596       | 1650       | Rationalism       |

There's some questions that comes to my mind right now, why don't use the philosopher name as the primary key? well, i answer that with another question. What happens if we have two philosophers with the same name? what happens if the philosopher changes his name?
what happens if we misspell a name? a lot of this can happen that would not identify correctly the particular philosopher that we're looking for. That's why, a good practice is use numbers as primary key. It only exists one number 1, and one number 2, so, there's no errors there.

**Another most common example**
```sql
CREATE TABLE users (
    id INTEGER PRIMARY KEY,
    name TEXT,
    email TEXT
);
```


|id|name|email|
|---|---|---|
|1|Alice|[alice@email.com](mailto:alice@email.com)|
|2|Bob|[bob@email.com](mailto:bob@email.com)|
- `id` is the **primary key**
- No two users can share the same `id`



# **Foreign Key**

We establish that a primary key, it's a key that gives identity to a particular row of a table. In the other side, a foreign key it's a column of a table that references the primary key of another table, creating a relationship between them. a connection between two separate entities.
Let's go back to the keys example showed before (the DNA, finger print and that.) in this context, those would be the primary keys, which would be the foreign keys?
	- Your parents (biological relationship)
	- Your teacher (knowledge relationship)
	- Your country (geographical relationship)
	- Your era (temporal relationship)

That's the same thing happening in data bases, we take a primary key from a table, and reference it into another one. Just like you parents DNA is referenced in yours.

**Example**
```sql

PRAGMA foreign_keys = ON;

CREATE TABLE books ( 
book_id INTEGER PRIMARY KEY, 
title TEXT NOT NULL, 
publication_year INTEGER, 
author_id INTEGER, 
FOREIGN KEY (author_id) REFERENCES philosophers(philosopher_id) 
); 

```

**What's happening here?** 
- `book_id` is the PRIMARY KEY (the book's unique identity) 
- `author_id` is a FOREIGN KEY that **points to** a philosopher's `philosopher_id`


**Philosopher table**

| philosopher_id | name      | birth_year | death_year | school_of_thought |
| -------------- | --------- | ---------- | ---------- | ----------------- |
| 1              | Plato     | -428       | -348       | Idealism          |

As you see, the author_id is the foreign key, that references the primary key from the philosophers table. 


#  **PRAXIS: How to Use Keys in SQLite**

### Creating a Primary Key

**Method 1: Explicit Declaration**
```sql
CREATE TABLE philosophers ( 
philosopher_id INTEGER PRIMARY KEY, 
name TEXT NOT NULL 
);
```

**Method 2: Composite Primary Key** (multiple columns together form uniqueness)
```sql
CREATE TABLE course_enrollment ( 
student_id INTEGER, 
course_id INTEGER, 
enrollment_date TEXT, 
PRIMARY KEY (student_id, course_id) 
);
```
This says: "The combination of student_id AND course_id must be unique" (a student can't enroll in the same course twice)

**Method 3: Auto-increment Primary Key**
```sql
CREATE TABLE philosophers ( 
philosopher_id INTEGER PRIMARY KEY AUTOINCREMENT, 
name TEXT NOT NULL 
);
```
SQLite will automatically assign 1, 2, 3, 4... as new rows are added.


## Creating a Foreign Key

⚠️ Important SQLite Detail (Very Important!

SQLite **does not enforce foreign keys by default** 😮

You MUST enable them:
```sql
PRAGMA foreign_keys = ON;
```
Otherwise:
- SQLite will **ignore foreign key rules**
- Invalid references will be allowed

```sql
PRAGMA foreign_keys = ON; 
CREATE TABLE books ( 
book_id INTEGER PRIMARY KEY, 
title TEXT NOT NULL, 
author_id INTEGER, 
FOREIGN KEY (author_id) REFERENCES philosophers(philosopher_id) 
);
```

|Concept|Meaning|
|---|---|
|Primary Key|Unique identity of a row|
|Foreign Key|Reference to another table|
|PK|“Who am I?”|
|FK|“Who do I belong to?”|
