Before talking about data types in SQL, lets talk about this same data types in the real world. We have numbers, to measure things, to make reality cuantificable.  We have words, that are composed of symbols, and together builds significance and meaning to the reality. We have truth or false values (yes or no, true or false, On or No, Good or bad.) And finally, we have the void, the emptiness.  Those values, together build the reality as we understand it. 
The same happens in SQL. SQLite is minimalist, it only have 5 different data types:


### INTEGER

This is very simple, and very similar to Python language.  These are entire number (not decimals.) It also allows entire negative numbers

**Examples**
	1
	34
	-45
	2026

**PRAXIS**
```sql

CREATE TABLE Philosophers (
	philosopher_id  INTEGER PRIMARY KEY,
	birth_year INTEGER,
	death_year INTEGER,
	number_of_books INTEGER,
);

INSERT INTO Philosophers VALUES (1, -428, -348, 36)
--This means plato id 1, bith year is -428n death year is -348 and wrote 36 books
```

**AXIOLOGY: Why It MATTERS**

Integers are:
- **Precise** (no rounding errors like decimals)
- **Fast** (computers process integers faster than decimals)
- **Ideal for IDs** (primary keys should be integers)
- **Natural for counting**


### REAL

Like in the real world we have entire and decimal numbers, also we have that in SQLite.
Why they are important? because there's certain type of data that can't be stored like a entire number. The temperature doesn't goes from 20 to 21 directly, it goes ascending in decimals. The weight of a person always have decimals. It's designed to store precise information in his numbers.

**Examples**
- `3.14159` (π)
- `-273.15` (absolute zero in Celsius, precise)
- `0.618` (golden ratio φ)
- `6.626e-34` (Planck's constant in scientific notation)

**PRAXIS**
```sql
CREATE TABLE meditation_sessions (
	session_id INTEGER PRIMARY KEY,
	duration_hours REAL,
	heart_rate_avg REAL,
	temperature_celsius REAL
	
);
INSERT INTO meditation_sessions VALUES (1, 1.5, 62.3, 36.8);
```

**AXIOLOGY: Why It MATTERS**

REAL numbers are:
- **Precise for measurements**
- **Essential for scientific calculation**
- **Natural for ratios and percentages**

**BUT - Warning:**
- Floating-point math has **rounding errors** (0.1 + 0.2 might equal 0.30000000000000004)
- Never use REAL for money (use INTEGER with cents)
- Not suitable for exact comparisons

### TEXT

This is very auto descriptive. Think of this, the 80% of the data available in the world, is made of text. Text is key in the data base, because the human nature is symbolic. We need symbols to give significance to our existence, we describe experiences in words, we communicate through language, we encode meaning on symbols. and this data type allow us to save text in our tables

**Examples**
- `'Plato'`
- `'The unexamined life is not worth living'`
- `'2025-02-06'` (dates as text)
- `'🙏'` (even emojis)


**PRAXIS**
```sql

CREATE TABLE wisdowm_quotes(
	quote_id INTEGER PRIMARY KEY,
	philosopher TEXT,
	quote TEXT,
	language TEXT,
);
INSERT INTO wisdom_quotes VALUES (1, 'Socrates', 'I only know, that i know nothing', 'English')
```

⚠️**Important:** Use single quotes `'text'` for string literals in SQL.



**AXIOLOGY: Why It MATTERS**
TEXT is:
- **Flexible** (any length, any characters)
- **Human-readable** (what you see is what you get)
- **Universal** (works with any language, any script)
- **Essential for communication**
**Key principle:** If humans need to READ it, store it as TEXT.



### BLOB

The BLOB type of data is particular, is not meant to be understand it by the human, is for the machine. His meaning is: BLOB = Binary Large Object. Stores raw binary data - **not human-readable**, but machine-processable. BLOB represents non textual information.

**Examples**
- Images (JPEG, PNG files)
- Audio files (MP3, WAV)
- Video files
- PDFs
- Encrypted data
- Serialized objects


**PRAXIS**
```python
import sqlite3 

# Store an image in the database 
with open('plato_portrait.jpg', 'rb') as file: 
	image_data = file.read() 
	

conn = sqlite3.connect('philosophy.db') 

cursor = conn.cursor() 

cursor.execute(""" 
CREATE TABLE philosopher_portraits ( 
philosopher_id INTEGER PRIMARY KEY, 
portrait BLOB 
) 
""") 

cursor.execute( 
"INSERT INTO philosopher_portraits VALUES (?, ?)", 
(1, image_data) 
)

conn.commit() conn.close()

```
**In practice:** Most developers store BLOBs externally (file system, cloud storage) and keep only the **path** as TEXT in the database.


**AXIOLOGY: Why It MATTERS**

BLOB is:

- **Versatile** (can store ANY type of file)
- **Integrated** (keeps media with related data)
- **Portable** (database contains everything)

**BUT - Consider:**

- BLOBs make databases **large**
- Harder to query (can't search inside an image)
- Many prefer storing files separately and keeping only **file paths** in the database

### NULL 

NULL represents the absence. Is not a zero, it doesn't mean empty. It means none. It sounds paradoxical, i know, but it's better illustrated with some examples.

NULL represents **unknown or inapplicable information**:
- **Unknown** - "What's Socrates' birth year?" → NULL (we don't know exactly)
- **Not applicable** - "What's Plato's death year?" (if he were alive) → NULL
- **Missing** - "What's the email address?" → NULL (not provided yet)
- **Undefined** - "What's the average of zero numbers?" → NULL (mathematically undefined)

Why null? well, because no all the data is known, not everything applies. Not everything exists yet. 

**PRAXIS**
```sql
CREATE TABLE Philosophers (
	philosopher_id INTEGER PRIMARY KEY,
	name TEXT NOT NULL,    --Cannot be null
	brith_year INTEGER,    --Can be Null, if we don't know it
	birth_death INTEGER    --Can be Null, if he still alive
	email TEXT,            --Can be Null, ancient philosophers didn't have email
);

INSERT INTO Philosophers VALUES (1, 'Plato', -428, -348, NULL);
--Plato has no email, obviosly.

-- Finding philosophers with unknown birth year 
SELECT name FROM philosophers WHERE birth_year IS NULL; 

-- Finding living philosophers (no death year) 
SELECT name FROM philosophers WHERE death_year IS NULL;
```

**AXIOLOGY: Why It MATTERS**

NULL allows:

- **Honesty** - "I don't know" instead of making up data
- **Flexibility** - optional fields don't need dummy values
- **Truth** - distinguishing between "zero" and "unknown"

**BUT - Warning:**

- NULL behaves strangely in comparisons
- `NULL = NULL` is **not true** (unknowable = unknowable → unknowable)
- Must use `IS NULL` or `IS NOT NULL` to check
