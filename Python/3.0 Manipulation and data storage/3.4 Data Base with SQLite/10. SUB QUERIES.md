# 🔍 Sub Queries

A subquery is a query living inside another query. Think of it as asking a question whose answer becomes the condition for a bigger question.

Imagine you want to find all employees who earn more than the average salary. You can't just write `WHERE salary > average` — you first have to *calculate* that average, and *then* use it as a filter. That two-step thought process is exactly what a subquery does for you, in a single SQL statement.

```sql
-- Mental model: "Find everyone who earns MORE than the average"
-- Step 1 (inner): calculate the average salary
-- Step 2 (outer): use that result as the filter threshold

SELECT name, salary
FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);
```

The database runs the inner query first, gets back a number, and then hands it to the outer query as if you had typed it by hand. Elegant.

---

# WHERE Subqueries — Filtering With a Question

This is the most common pattern. The inner query returns a single value (or a list), and the outer query uses it to filter rows.

```sql
-- Single value: who earns the most?
SELECT name, salary
FROM employees
WHERE salary = (SELECT MAX(salary) FROM employees);


-- List of values: which employees are in the "Engineering" department?
-- Mental model: first find the dept_id for Engineering, then find those employees
SELECT name
FROM employees
WHERE department_id IN (
    SELECT id
    FROM departments
    WHERE dept_name = 'Engineering'
);
```

`IN` is your best friend when the subquery returns multiple rows. It says: "is this value *anywhere* in that list?"

# FROM Subqueries — A Temporary Table

You can also place a subquery in the `FROM` clause. This creates what SQL calls a **derived table** — a temporary result set that exists only for the duration of your query.

```sql
-- Mental model: create a summary table first, then query it
SELECT dept_name, avg_salary
FROM (
    SELECT
        departments.dept_name,
        AVG(employees.salary) AS avg_salary
    FROM employees
    JOIN departments ON employees.department_id = departments.id
    GROUP BY departments.dept_name
) AS department_summary
WHERE avg_salary > 50000;
```

Notice the `AS department_summary` — you must give your derived table an alias. SQL requires it.

# Correlated Subqueries — The Loop Inside a Query

A **correlated subquery** is one that references a column from the *outer* query. This makes it run once per row of the outer query — like a loop.

```sql
-- Mental model: "For each employee, check if their salary is above
-- the average salary of *their own department*"
SELECT name, salary, department_id
FROM employees AS e
WHERE salary > (
    SELECT AVG(salary)
    FROM employees
    WHERE department_id = e.department_id  -- references outer query!
);
```

The `e.department_id` is the bridge between outer and inner. For each row the outer query examines, the inner query recalculates with that specific department. Powerful — and a little slower for very large tables, so use it with intention.

# EXISTS — Does Anything Match?

`EXISTS` checks whether a subquery returns *any* rows at all. It doesn't care about the values — only about presence.

```sql
-- Find all customers who have placed at least one order
SELECT name
FROM customers AS c
WHERE EXISTS (
    SELECT 1
    FROM orders
    WHERE orders.customer_id = c.id
);
```

The `SELECT 1` inside is conventional — you're not interested in what it returns, only *whether* it returns anything.

# Full Python Example with SQLite

```python
import sqlite3

with sqlite3.connect('empresa.db') as conn:
    cursor = conn.cursor()

    # -------------------------------------------------------
    # EXAMPLE 1: Employees earning above average
    # -------------------------------------------------------
    print("=" * 55)
    print("Employees Earning Above Average")
    print("=" * 55)

    cursor.execute("""
        SELECT name, salary
        FROM employees
        WHERE salary > (SELECT AVG(salary) FROM employees)
        ORDER BY salary DESC
    """)

    for name, salary in cursor.fetchall():
        print(f"  {name:<20} ${salary:,.2f}")


    # -------------------------------------------------------
    # EXAMPLE 2: Departments with above-average team salary
    # -------------------------------------------------------
    print("\n" + "=" * 55)
    print("High-Performing Departments (avg salary > $60k)")
    print("=" * 55)

    cursor.execute("""
        SELECT dept_name, avg_salary
        FROM (
            SELECT
                d.dept_name,
                AVG(e.salary) AS avg_salary
            FROM employees e
            JOIN departments d ON e.department_id = d.id
            GROUP BY d.dept_name
        ) AS summary
        WHERE avg_salary > 60000
        ORDER BY avg_salary DESC
    """)

    for dept, avg in cursor.fetchall():
        print(f"  {dept:<25} ${avg:,.2f}")
```

# 🧩 Quick Reference

```sql
-- Subquery returning a single value (scalar)
WHERE column = (SELECT single_value FROM ...)

-- Subquery returning multiple values (list)
WHERE column IN (SELECT many_values FROM ...)

-- Subquery as a derived table
FROM (SELECT ... FROM ...) AS alias_name

-- Correlated subquery (runs once per outer row)
WHERE column > (SELECT ... WHERE inner.col = outer.col)

-- Existence check
WHERE EXISTS (SELECT 1 FROM ... WHERE ...)
```

The deeper you go into SQL, the more you'll find that subqueries are not a shortcut — they are a way of thinking in layers. Master the inner query first. Then ask the outer question.
