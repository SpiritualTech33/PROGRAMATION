# 🗂️ Queue — IO-Bound vs CPU-Bound

We've talked about the philosophy of concurrency and parallelism. We've used `asyncio` for waiting intelligently and `multiprocessing` for true parallel computation. Now we need to talk about the **dispatcher** that organizes work: the Queue.

And more importantly, we need to be precise about something that every professional Python developer must internalize: the difference between **IO-bound** and **CPU-bound** tasks — because the right tool depends entirely on knowing which one you're dealing with.

---

# IO-Bound vs CPU-Bound — The Core Distinction

Think of a chef in a kitchen.

**IO-bound:** The chef sends an order to the supplier and *waits for the delivery*. While waiting, they are idle — doing nothing. The bottleneck is not the chef's skill, it's the waiting time for external responses. This is network requests, file reads, database queries. The CPU is mostly idle, blocked on input/output.

**CPU-bound:** The chef is grinding spices by hand. They are working at 100% capacity the entire time. The bottleneck is the raw processing power. This is numerical computation, image processing, machine learning training. The CPU is maxed out.

```
IO-Bound Tasks                  CPU-Bound Tasks
─────────────────────           ─────────────────────
Web API requests                Matrix multiplication
Database queries                Image/video processing
File reads & writes             Compression algorithms
Network communication           Machine learning inference
Waiting for user input          Cryptographic operations

Best tool: asyncio              Best tool: multiprocessing
(concurrent, single thread)     (parallel, multiple processes)
```

Why does this distinction matter so much? Because using `asyncio` for a CPU-bound task gives you zero benefit — you're still stuck with one Python thread doing heavy computation. And spawning dozens of processes for IO-bound tasks is wasteful overkill when one thread switching intelligently would do the job better.

---

# The Queue — Managing a Flow of Work

A Queue is a data structure built on a simple philosophy: **First In, First Out (FIFO)**. Items go in one end, come out the other. Think of it as a conveyor belt — producers place items on it, consumers take them off.

It becomes essential when you have multiple workers processing a stream of tasks, and you need to coordinate them without race conditions or missed work.

## asyncio.Queue — For IO-Bound Work

```python
import asyncio
import aiohttp

# -------------------------------------------------------
# Producer: generates tasks and puts them in the queue
# -------------------------------------------------------
async def url_producer(queue, list_of_urls):
    """
    Mental model: A dispatcher placing work orders on a conveyor belt.
    """
    for url in list_of_urls:
        await queue.put(url)
        print(f"[Producer] Queued: {url}")

    # Poison pills — signal workers to stop
    for _ in range(NUM_WORKERS):
        await queue.put(None)


# -------------------------------------------------------
# Consumer: takes tasks from the queue and processes them
# -------------------------------------------------------
async def url_consumer(worker_id, queue, session):
    """
    Mental model: A worker picking tasks off the conveyor belt.
    Stops when it receives the None sentinel.
    """
    while True:
        url = await queue.get()

        if url is None:
            print(f"[Worker {worker_id}] No more work. Shutting down.")
            queue.task_done()
            break

        try:
            async with session.get(url) as response:
                data = await response.json()
                print(f"[Worker {worker_id}] Fetched: {url} → status {response.status}")
        except Exception as e:
            print(f"[Worker {worker_id}] Failed: {url} → {e}")
        finally:
            queue.task_done()


# -------------------------------------------------------
# Orchestrator: wires everything together
# -------------------------------------------------------
NUM_WORKERS = 3

async def main():
    urls = [
        "https://jsonplaceholder.typicode.com/posts/1",
        "https://jsonplaceholder.typicode.com/posts/2",
        "https://jsonplaceholder.typicode.com/posts/3",
        "https://jsonplaceholder.typicode.com/posts/4",
        "https://jsonplaceholder.typicode.com/posts/5",
    ]

    queue = asyncio.Queue(maxsize=10)

    async with aiohttp.ClientSession() as session:
        # Start producer and workers concurrently
        await asyncio.gather(
            url_producer(queue, urls),
            *[url_consumer(i, queue, session) for i in range(NUM_WORKERS)]
        )

asyncio.run(main())
```

The `queue.task_done()` call is important — it tells the queue that a task has been processed. If you later call `await queue.join()`, it will block until all tasks are marked done.

## multiprocessing.Queue — For CPU-Bound Work

When the work is computationally heavy, you need real parallelism across CPU cores. `multiprocessing.Queue` is thread- and process-safe, built for exactly this.

```python
import multiprocessing
import numpy as np
import time

# -------------------------------------------------------
# Worker function — runs in a separate process
# -------------------------------------------------------
def compute_worker(worker_id, task_queue, result_queue):
    """
    Mental model: A separate brain (process) with its own memory.
    Picks up computation tasks and sends results back.
    """
    while True:
        task = task_queue.get()

        if task is None:  # Poison pill — time to stop
            print(f"[Worker {worker_id}] Shutting down.")
            break

        matrix_size, task_id = task
        print(f"[Worker {worker_id}] Computing task {task_id}...")

        # CPU-intensive work: matrix multiplication
        A = np.random.rand(matrix_size, matrix_size)
        B = np.random.rand(matrix_size, matrix_size)
        result = np.dot(A, B)

        result_queue.put({
            'task_id': task_id,
            'worker': worker_id,
            'result_shape': result.shape,
            'checksum': float(result.sum())
        })


# -------------------------------------------------------
# Main process: dispatches tasks and collects results
# -------------------------------------------------------
def main():
    NUM_WORKERS = multiprocessing.cpu_count()
    NUM_TASKS   = 10
    MATRIX_SIZE = 300

    task_queue   = multiprocessing.Queue()
    result_queue = multiprocessing.Queue()

    # Start worker processes
    workers = [
        multiprocessing.Process(
            target=compute_worker,
            args=(i, task_queue, result_queue)
        )
        for i in range(NUM_WORKERS)
    ]
    for w in workers:
        w.start()

    # Send tasks
    for i in range(NUM_TASKS):
        task_queue.put((MATRIX_SIZE, i))

    # Send poison pills to stop workers
    for _ in range(NUM_WORKERS):
        task_queue.put(None)

    # Collect results
    results = []
    for _ in range(NUM_TASKS):
        results.append(result_queue.get())

    # Wait for all workers to finish
    for w in workers:
        w.join()

    print(f"\nCompleted {len(results)} computations.")
    for r in sorted(results, key=lambda x: x['task_id']):
        print(f"  Task {r['task_id']:>2} | Worker {r['worker']} | checksum: {r['checksum']:.2f}")


if __name__ == '__main__':
    main()
```

# The Decision Map

```
You have a task to parallelize.
         │
         ▼
Is the bottleneck WAITING (network, disk, DB)?
         │
    YES ─┼─ NO (the bottleneck is raw computation)
         │           │
         ▼           ▼
    asyncio      multiprocessing
    + asyncio.Queue  + multiprocessing.Queue
    (concurrent,     (truly parallel,
     single thread,   multiple processes,
     low overhead)    high overhead, high power)
```

# 🧩 Quick Reference

```python
# asyncio.Queue (IO-bound)
queue = asyncio.Queue(maxsize=N)
await queue.put(item)          # add item (waits if full)
item = await queue.get()       # remove item (waits if empty)
queue.task_done()              # signal task is complete
await queue.join()             # wait until all tasks are done

# multiprocessing.Queue (CPU-bound)
queue = multiprocessing.Queue()
queue.put(item)                # add item (blocks if full)
item = queue.get()             # remove item (blocks if empty)

# Poison pill pattern (universal stop signal)
queue.put(None)                # one None per worker
# Inside worker:
if task is None: break
```

The Queue is the bridge between producing work and consuming it. Master this pattern and you have the foundation for building any data pipeline, web scraper, batch processor, or ML training system that needs to scale.
