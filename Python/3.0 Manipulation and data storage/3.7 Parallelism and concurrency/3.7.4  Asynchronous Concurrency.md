Let's make this very clear from the beginning to avoid any type of confusions. Concurrency is not parallelism, are two different things, that seems to work the same, but they're working differently.  Remember what you just learned before? about how the threads are designed to be good at waiting? well, is above this idea were asynchronous concept steps.
	Parallelism says: _"Let's do two things at the exact same instant."_ Concurrency says: _"Let's manage multiple things so intelligently that waiting time becomes productive time."_
	One is about **simultaneous execution.** The other is about **intelligent time management.**

Key idea:
	While one task is waiting, do something else.


I think it's very clear the main differences between parallelism and concurrency, but let's use a metaphor to make it cristal clear. 

Imagine two coffee shop baristas, lets call them Paola and Carolina (P for parallelism and C for concurrency)

Let's begin with Paola, she takes the order of client A, a Mokka latte, goes to the coffee machine, makes the coffee, waits for the coffee to be ready and goes back to the client A, gives him the coffee and then goes with the client B, asks for his order and repeats the same process.

In the other side, we have Carolina, she is a little bit more energetic than Paola, and is smarter too. She takes the order from customer A, goes to the coffee machine and start serving the coffee, meanwhile she waits for the coffee, se goes back to the bar, and asks the customer B order, takes it and goes back to the machine and check if customer A coffee is already done, in this case, is still unfinished, so she goes back to the bar and asks client C for his order, so she goes back to the machine, she realizes that customer A coffee is ready, so she take it and hands it to customer A with a smile, and repeat the same process to customer B and C.

With this description, witch scenario you think is more efficient with waiting times?

Paola's visual process:
Serve customer A → wait 4 min for espresso → deliver → 
Serve customer B → wait 4 min for espresso → deliver →
Serve customer C → wait 4 min for espresso → deliver
Total: 12 minutes. 8 of those minutes: pure waiting.

Carolina's visual process:
Start espresso for A → WHILE WAITING → take order from B →
Start espresso for B → WHILE WAITING → take order from C →
Espresso A ready → deliver to A → Espresso B ready → deliver to B...
Total: ~5 minutes. Zero wasted waiting.

Carolina is your **single Python thread** running an async program.


# The event loop - The Brain of Concurrency
---
Think of this like a coordinator with limitless patience, constantly asking if any task is done. Visually would be something like this:

"Is Task A done waiting? No? Okay.
 Is Task B done waiting? Yes! Run it.
 Is Task C done waiting? No? Okay.
 Is Task A done waiting? No? Okay.
 Is Task B done waiting? No? Okay.
 Is Task C done waiting? Yes! Run it.
 ..."
It never sleeps. It never stops checking. And it switches between tasks **only at moments of waiting** — never interrupting active computation.
