A processor with multiple independent cores that can work simultaneously in different tasks. This is new raw power, that allows us to do more in less time. This is great, now we can make more operations in less time, but there's one problem related with Python, and it's the GIL (Global Interpreter Lock.) What's this? GIL sounds like the word used in Argentina to refer to a young man or a friend, but in Python context means a whole different thing. 
Python was born in a single core era, so it wasn't really created to support multiprocessing tasks. Python manages memory automatically, i don't know the reasons of this decision, but i can intuit that it was designed this way to make the process of coding easier. Languages like C allows you to determinate the memory position of you variables and that stuff, but is way more harder to manage manually the memory. But with this decision, you don't have really the chance of decide  how to manage the memory. So Python does it for you. The problem is, that it doesn't matter if you have a NASA computer with a processor with 48 cores and the last tech in reading speed, Python will execute a single Thread at the time. Python makes this to prevent memory corruption, simplify garbage collection, make single threads performance faster, and make c extensions easy to write. So, if this is true, if we can only make a single thread at the time using Python, what's the point of having multiple cores? well, don't worry pall, because there's a library for it.

I introduce to you the multiprocessing module from Python. A module that allows you to really use each core of your computer and use those juicy 48 cores from your NASA Pc jaja. But, how it does it? well, the multiprocessing module create separate processes for each core. Each process with his own memory space, his own Python interpreter, his own GIL and each process assigned to one physical core. Giving us as a result True parallelism, we're now doing multiple tasks at the same time. It's not amazing? 
The GIL is a constraint at the interpreter level.  
Multiprocessing is a strategy to transcend that constraint.

## The Three Dimensions

**Physical** — Multi-core chips are literal silicon architecture. Multiple execution units etched into the same wafer, sharing cache memory but capable of independent computation. When you use `multiprocessing`, you're directly mapping your code to this hardware reality.

**Mental** — The GIL is a philosophical trade-off. Safety over speed. Guido chose a world where your code _never_ corrupts memory over a world where it _might_ run faster. Every architectural decision is a value judgment.

**Spiritual** — There's something profound here. The GIL is essentially saying: _"one consciousness at a time."_ And multiprocessing responds: _"then we'll create multiple consciousnesses."_ This mirrors a deep question in philosophy of mind — can awareness be truly parallel, or does it always serialize experience?
